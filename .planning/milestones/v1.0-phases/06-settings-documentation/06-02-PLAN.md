  <name>Task 1: Create SECURITY.md with risk analysis and hardening tips</name>
  <files>SECURITY.md</files>
  <action>
    Create SECURITY.md in project root with the following structure and tone (honest + practical, not alarmist, not dismissive):

    1. **Header** -- "Security" title with brief intro explaining the document's purpose

    2. **Risk Summary** -- Table with columns: Risk | Severity | Mitigation | Status
       Required rows (per locked decisions):
       - Container escape | Low | Dropped capabilities (MKNOD, AUDIT_WRITE, SETFCAP, SETPCAP, NET_BIND_SERVICE, SYS_CHROOT, FSETID), no-new-privileges, resource limits | 🟢 Hardened
       - Credential exposure | Medium | .env gitignored, runtime injection only via docker -e, never baked into image layers | 🟢 Hardened
       - Network access from container | Medium | Containers have outbound internet access (intentional -- Claude Code needs it for API calls and package installs) | 🟡 User responsibility
       - Resource abuse | Low | Memory (configurable, default 2g) and CPU (configurable, default 2) limits enforced via docker flags | 🟢 Hardened
       - AI agent permissions (--dangerously-skip-permissions) | Medium | Container isolation is what makes this trade-off acceptable -- the agent has full filesystem access inside the container, but the container itself is sandboxed from the host | 🟡 User responsibility
       Add additional risks at discretion (e.g., SSH key management, volume mount exposure, image supply chain)

    3. **Docker Desktop vs Docker Engine** -- Brief section explaining:
       - Docker Desktop runs containers inside a lightweight VM (additional isolation layer)
       - Docker Engine on Linux runs containers directly on host kernel (still isolated via namespaces/cgroups but no VM boundary)
       - Recommendation: Docker Desktop provides stronger isolation for security-sensitive use cases

    4. **What We Do** -- Bullet list of all hardening measures applied (capabilities dropped, localhost-only SSH, key-only auth, no root password, resource limits, credential injection pattern)

    5. **What's Your Responsibility** -- Bullet list: rotate API keys, review container logs, keep Docker updated, understand --dangerously-skip-permissions implications, don't store sensitive files in workspace mount

    6. **Hardening Tips** -- Practical tips: rotate API keys periodically, use Docker Desktop for VM isolation, review workspace contents before pushing to git, set conservative resource limits, monitor container network activity if concerned

    7. **Reporting Issues** -- Brief: "Open a GitHub issue" (no formal responsible disclosure per locked decision)
  </action>
